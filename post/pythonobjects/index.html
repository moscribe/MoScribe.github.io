[编程随笔]Python--面向对象的编程

2020-02-10

文章目录

  • 无用之用
  • 当有大用

在Python的代码中, 你所看到的每一个字符串, 都是对象.

无用之用

指向一切的指针

在Cpython的代码中, 最常见的一个指针就是PyObject*, 这个指针是定义在Include/object.h中的一个结构体:

1
2
3
4
5

typedef struct _object {
    _PyObject_HEAD_EXTRA
    Py_ssize_t ob_refcnt;
    struct _typeobject *ob_type;
} PyObject;

这段代码的意思也很明确, 先是头部初始化, 然后一个引用计数器, 最后一个结构体指针比较复杂, 它表示这个结构体所拥有的类型, 这个结构体中包含了一个字符串, 几个整数以及一大堆的函数指针, CPython的动态类型系统就是在运行时通过该结构体判断类型并通过结构体中的函数指针完成各种操作的.

不过最为有趣的, 是PyObject的注释:

Nothing is actually declared to be a PyObject, but every pointer to a Python object can be cast to a PyObject*. This is inheritance built by hand. Similarly every pointer to a variable-size Python object can, in addition, be cast to PyVarObject*.

上来就告诉我不会有直接定义成PyObject* 的东西, 但每一个指向Python对象的指针都可以被转换为PyObject*. 后面一句话更为神奇: 这是手工实现的继承.

不定结构

到现在为止, 这个结构体功能上是不完整的. 因为这个结构体在内存中是定长的, 但是Python中存在大量变长结构, 光用一个结构体是无法于Python中的数据结构意义对应的, 也就是说我们需要一个结构体, 能管理Python中的变长结构. 当然这里的变长并不是让结构体变长, 在C语言里面可以使用指针指向变长的内存区域, 通过内存管理函数控制内存的大小; 与此同时, 结构体头部需要记录变长数据区的大小, 因此就有了PyVarObject这个结构体:

1
2
3
4

typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size; /* Number of items in variable part */
} PyVarObject;

这个结构体以PyObject开头, 只是在后面加了一个记录内存区域大小的整数. 按照上面注释的说法, 这个结构体"继承"自PyObject, 所有适用于PyObject的操作均适用于PyVarObject. 于是我们就有了可以同时管理定长于变长内存区域的结构体了.

一些宏

1
2
3
4

#define PyObject_HEAD                   PyObject ob_base;
#define PyObject_VAR_HEAD               PyVarObject ob_base;
#define _PyObject_CAST(op)              ((PyObject*)(op))
#define _PyVarObject_CAST(op)           ((PyVarObject*)(op))

前两个宏用在声明各种Object的时候, 放在结构体的头部, 这样以来每个结构体所代表的类型都可以用*PyObject* *指向其内存区域. 后两个宏用于显式类型转换.

无处不在的结构

看遍CPython的代码就会发现, 虽然代码里面充斥着无数的PyObject, 但是从来没有一个变量或者结构是直接用它声明或者定义的. PyObject这个结构体存在于每个对象内存区的开头的位置, 从表面上看来, 它不是任何一个对象, 但实际上, PyObject是每一个对象.

先看一个简单的列表插入的函数:

4

int
PyList_Insert(PyObject *op, Py_ssize_t where, PyObject *newitem)
{
    if (!PyList_Check(op)) {
        PyErr_BadInternalCall();
        return -1;
    }
    return ins1((PyListObject *)op, where, newitem);
}

这个函数接受的参数里面不管时待插入的元素newitem还是要插入元素的列表op都是以PyObject指针的形式传入的, 先检查是否为列表结构,然后才会将这个结构指针显式的转换为列表的对象指针并进行一系列的操作. 像这样的函数在CPython中比比皆是, 作为模块的API, 这些函数的参数都是以PyObject的形式传入的, 在处理之前都必须要经过类型的检查才能调用模块内相应的函数.

通过将所有的对象管理的指针全都变成PyObject, CPython实现了语言层面对"对象"这一概念的抽象化.

typedef struct {
    PyObject_HEAD
    double ob_fval;

typedef struct {
    PyObject_VAR_HEAD
    /* Vector of pointers to list elements.  list[0] is ob_item[0], etc. */
    PyObject **ob_item;

    /* ob_item contains space for 'allocated' elements.  The number
     * currently in use is ob_size.
     * Invariants:
     *     0 <= ob_size <= allocated
     *     len(list) == ob_size
     *     ob_item == NULL implies ob_size == allocated == 0
     * list.sort() temporarily sets allocated to -1 to detect mutations.
     *
     * Items must normally not be NULL, except during construction when
     * the list is not yet visible outside the function that builds it.
     */
    Py_ssize_t allocated;
} PyListObject;

第一个结构体是Python里面的浮点数, 值部分就是C语言里面的double. 这是一个典型的定长结构, 所以头部使用的是PyObject_HEAD这个宏. 第二个结构体是Python里面用到最多的list结构, 是一个典型的变长结构, 所以头部使用的是PyObject_VAR_HEAD这个宏. 这是两个截然不同的类型的结构体, 但是它们的开头都是一个PyObject的结构, 这个结构在所有的表示类型的结构体中都位于开头的位置, 因此任意类型的指针都可以被显式或隐式的转换成PyObject*的指针, 同时任何PyObject*的指针也可以根据PyObject结构体中的ob_type指向的类型被显式或隐式的转换成对应的类型.

这是一个很长的结构体, 这里只展现个大概. 这个结构体里面定义了一个对象在运行时所有可能的操作, 如果是列表对象, 就会一个由该结构体定义的一个PyList_Type, 在运行时会创建这样一个PyList_Type的结构体并一直存在于内存的某一个地方, 类型检查器会检查内存首部的指针是否和PyList_Type的地址匹配, 也就是运行时的类型检查, 比如上面的PyList_New函数中的*PyList_Check()*宏就是一个类型检查器.

以上所说的过程就是Python中实现动态类型的基础. 在Python的代码中, 所有解析出的符号和语法单元都有一个对象指针直接管理. 在Python运行的时候才会检查对象的类型, 将其指针转换成对应类型的指针, 然后才可以调用对象的值和函数. 这一设计在很大程度上导致了Python的运行缓慢(另一个原因是数据运行在堆上).

这一设计也体现了写Python代码时的一个基本思想: 一切皆是对象, 无论是一个数, 一个字符串, 一个列表还是一个函数, 在内存里面它们都只是一段内存上的数据, 一个由PyObject指针管理的对象. 这也是我们在学习Python的时候并不会像学习Java一样纠结于何为对象, 而更多的关注代码的逻辑本身. 当一切都是对象的时候, 其实就可以不用管什么是对象了, 专注于逻辑本身才是编程的本意.

大道至繁, 大道至简

我所接触的第一门编程语言是Java, 没有其它原因, 只是因为当初它的名气够大. Java中有各种繁复的概念, 现在看来基本都是这几十年的面向对象编程的积累. 与C++不同, Java的概念并不是为了榨取机器的性能, 而是一种经验上的总结. Java 是大道至繁的典型代表, 是工业时代的产物.

何为对象? 对象就是一段计算机内存数据的所有权. 何为继承? 继承就是内存区的复合. 我自己是比较喜欢这个解释, 因为相比于各种Java书中的冗长的解释, 这个解释更加地接近数学. 对象这个概念可以说是数学中的集合在编程中的体现. 现代数学的基石是集合, 处理的基本单元也是集合. 要把集合这个概念从数学的抽象变成计算机内存中的具体数据并不是一件简单的事情, Python的设计则是一种尝试, 这个设计继承自Lisp. 在Lisp中, 表这一结构就是数学集合的抽象, 或者说是集合的延伸. 在Python中, 不仅仅是将所有写下的符号通通看成是对象, 而且对象可以是由对象组成的. 一旦一个数学上的概念可以开始递归, 那这个概念就有了近乎无限的变化可能.

大道至繁, 是我们总是想用一个工具解决所有问题, 所以才会越来越复杂.

大道至简, 则是在认识到了一定程度的时候, 找出共同的地方; 研究抽象的共同点, 会让事情变得简单. Python是这一理念的践行者, 但是Python既不是做的最好的, 也不是走的最远的, Python只是最受欢迎的那一个.
